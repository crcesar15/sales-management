### Suggestion: Show Page as a Tabbed Layout

Consider organizing the Show page with two sections:
- **Details tab**: product info, images, brand, categories, status
- **Variants tab**: options panel + variants panel (Task 03 content)

This keeps the page clean and lets users focus on one aspect at a time. If the product is simple (one variant, no options), the variants tab can show a simplified view with just the default variant's price/stock.

## Key Inertia Patterns

### Props (from controller)

| Prop | Shape |
|------|-------|
| `product` | Full resource: id, name, description, status, brand, categories, media (with thumb/full URLs), default_variant (if simple) |
| `product.options` | `Array<{ id, name, values: Array<{ id, value }> }>` |
| `product.variants` | `Array<{ id, identifier, price, stock, status, option_values: Array<{ id, value, option_name }>, images: Array<{ id, thumb_url }> }>` |

### All Actions Redirect Back

Options CRUD, variant CRUD, and image sync all use `router.post/put/delete` and redirect back to the product show page. On success, Inertia refreshes the props automatically. On validation error, `setErrors()` populates field errors. On guard failure (e.g., deleting an in-use option value), the flash error shows via toast.

## OptionsPanel

```
[+ Add Option]
  Color          [+ Add Value]
    • Red  [×]
    • Blue [×]
  Size           [+ Add Value]
    • S [×]  • M [×]  • L [×]

[Generate Variants]  ← shown only when product has default variant with no options
[+ Add Variant]      ← always available
```

- **Add option**: opens inline input or small dialog, submits `POST /products/{product}/options`
- **Rename option**: inline edit on double-click or edit icon, submits `PUT /products/{product}/options/{option}`
- **Delete option**: confirm dialog, submits `DELETE /products/{product}/options/{option}` — cascade deletes values
- **Add value**: inline input under the option, submits `POST /products/{product}/options/{option}/values`
- **Delete value**: "×" button, submits `DELETE /products/{product}/options/{option}/values/{value}` — returns 422 if used by a variant (show flash error toast)
- **Generate**: opens `AutoGenerateDialog`

## VariantsPanel

DataTable columns: Identifier | Options (chips) | Price | Stock | Status | Images | Actions

| Column | Behavior |
|--------|----------|
| Identifier | Editable text. Nullable — show "—" if empty. |
| Options | Chips/tags per option value (e.g., `Color: Red`, `Size: M`). For default variant with no options, show "Default variant" badge. |
| Price | Editable. Currency formatting (`InputNumber` with `mode="currency"`). |
| Stock | **Read-only**. Show current value with a tooltip "Managed by inventory". Only set during creation. |
| Status | `Tag` component — active (success), inactive (warn), archived (danger). Editable dropdown in edit mode. |
| Images | Small thumbnail strip. Click opens `VariantImagePicker`. |
| Actions | Edit (inline or dialog), Delete (with confirm). |

### Inline Edit vs Dialog Edit

For the variant table, inline editing (click to edit a cell) is smoother for single-field changes like price or status. For multi-field edits (identifier + price + status + option values), use `ManualVariantDialog` in edit mode.

### Suggestion: Default Variant Row Highlight

The default variant (the one auto-created with the product) should be visually distinct — a subtle background highlight or a "Default" badge. This helps users identify it during the "keep & edit" transition.

## AutoGenerateDialog

### Step 1: Define Options and Values

Dynamic form where user adds options (e.g., "Color") and their values (e.g., "Red", "Blue"). Uses reactive state (not submitted until step 2).

### Step 2: Preview Combinations

Computed count: `product of all value counts` (e.g., 2 colors × 3 sizes = 6 variants).

**"Keep & edit" awareness**: when the product has only the default variant, show a notice: *"The existing variant will be assigned to the first combination. You can change which combination it maps to below."*

Show the combinations list with the first one tagged as "Existing variant →" and the rest tagged as "New".

### Step 3: Confirm and Generate

Submits `POST /products/{product}/variants/generate` with the options data. On success, dialog closes and the variants panel refreshes with the new variants.

### Availability Rules

- **Enabled** when product has exactly one variant (the default) with zero option values
- **Disabled** when product already has multiple variants or options with assigned values
- The "Generate Variants" button in `OptionsPanel` should be hidden/disabled based on these rules

## ManualVariantDialog

Form with fields: Identifier (optional), Price (required), Stock (required, only on create), Status, Option Values (multi-select chips — one value per option).

- **Option values**: for each option defined on the product, show a dropdown of its values. User picks one per option. Validates no duplicate combination exists before submit (server-side check, show 422 error).
- **Edit mode**: pre-fills from existing variant data. Stock field is read-only with "Managed by inventory" tooltip.

## VariantImagePicker

A dialog or panel that shows the product's image gallery as a selectable grid.

### Suggestion: Toggle-Based Image Grid

Display product images as a thumbnail grid. Each image has a toggle (checkbox overlay or click-to-select). Selected images get a checkmark overlay. This feels natural for "pick which photos represent this variant."

### Behavior

- Loads the product's images from props
- Pre-selects images already associated with the variant (from `variant.images`)
- User toggles images on/off
- On confirm: submits `PUT /products/{product}/variants/{variant}/images` with the selected `media_ids`
- **Inheritance indicator**: if no images are explicitly selected, show a note "This variant will show all product images"

### Suggestion: Quick Color/Visual Tagging

For products with many images (e.g., clothing), consider a drag-to-group interaction where users drag images onto variant cards. But the toggle grid is simpler to implement and should be the starting point.

## Gotchas

1. **Combination count preview** must update reactively as user adds/removes values in `AutoGenerateDialog` — use a `computed` that multiplies all value array lengths.
2. **Disable "Generate"** if any option has zero values or if product already has variants beyond the default.
3. **Stock is read-only in edit.** Show current value. Tooltip: "Managed by inventory". Only settable during manual variant creation or auto-generate (defaults to 0).
4. **"Keep & edit" transition** — the `AutoGenerateDialog` should clearly show which combination maps to the existing default variant. The user can change this mapping before confirming.
5. **Image inheritance** — if a variant has no explicit image associations, show all product images in the variant's thumbnail strip with a subtle "inherited" indicator. The `VariantImagePicker` shows a note: "No images selected — showing all product images."
6. **No soft delete on variants** — the delete action is permanent. Use a clear confirmation message: "This variant will be permanently deleted."
7. **Variant deletion hard-deletes pivot rows** — no orphaned data, but no undo either.
8. **`v-can` directive** — variant actions reuse product permissions: `v-can="'product.edit'"` for create/update, `v-can="'product.delete'"` for delete.
